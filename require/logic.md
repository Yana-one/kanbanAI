# AI 칸반보드 업무 자동 분류 프롬프트 로직 (Gemini API 용)

이 문서는 사용자가 입력한 "할 일(Task)" 데이터를 Gemini API가 어떻게 분석하고 분류해야 하는지에 대한 명확한 기준과 프롬프트 템플릿을 정의합니다. `require/database.md`, `require/borad.md`, `require/prd.md`를 기반으로 작성되었습니다.

---

## 1. AI 분류의 핵심 목표

사용자가 입력한 자유 형식의 업무 제목(Title) 및 설명(Description)을 분석하여 아래 세 가지 필드를 자동 결정합니다.
1. **카테고리 (`category`)**: 업무의 성격이나 부서를 나타내는 문자열.
2. **우선순위 (`priority`)**: 업무의 중요도 및 긴급도 (`high`, `medium`, `low`).
3. **상태 (`status`)**: 업무의 현재 진행 상태 (`todo`, `in_progress`, `done`).

---

## 2. 분류 기준 (Classification Logic)

### 2-1. 카테고리 (`category`) 기준
업무의 문맥과 사용된 단어를 바탕으로 적절한 직무/분야 카테고리로 매핑합니다.
*   **개발**: "API", "서버", "버그", "프론트엔드", "데이터베이스", "테스트", "코드", "배포" 등.
*   **디자인**: "디자인", "UI", "UX", "피그마", "시안", "목업", "로고", "배너" 등.
*   **마케팅**: "광고", "캠페인", "SNS", "인스타그램", "프로모션", "이벤트", "카피라이팅" 등.
*   **기획**: "기획안", "PRD", "요구사항", "리서치", "일정", "와이어프레임" 등.
*   **운영**: "문서", "가이드", "고객응대", "CS", "결제", "정산" 등.
*   **기타**: 위의 분류에 명확하게 들어맞지 않는 일반적인 업무.

### 2-2. 우선순위 (`priority`) 기준
일반적으로 `medium`을 기본값으로 하되, 긴급성과 중요도를 암시하는 단어가 있을 경우 조정합니다.
*   **`high` (높음)**: "긴급", "ASAP", "오늘까지", "치명적", "크리티컬", "핫픽스", "장애", "최우선" 등의 키워드가 포함된 경우.
*   **`medium` (보통)**: 일반적인 업무로, 특별히 긴급함을 나타내는 단어가 없는 경우 (기본값).
*   **`low` (낮음)**: "나중에", "시간 날 때", "여유있을 때", "아이디어", "백로그", "참고용" 등의 키워드가 포함된 경우.

### 2-3. 상태 (`status`) 기준
기본적으로 새로 생성된 업무이므로 `todo`로 설정하는 것이 일반적이나, 텍스트에 진행 상황이 명시된 경우 이를 반영합니다.
*   **`todo` (할 일)**: "할 예정", "시작하기", "해야 함", 또는 별다른 상태 언급이 없는 일반적인 업무 (기본값).
*   **`in_progress` (진행 중)**: "진행 중", "작성 중", "검토 중", "수정 중" 등 현재 작업이 이루어지고 있음을 암시하는 경우.
*   **`done` (완료)**: "완료됨", "끝남", "배포 완료", "처리 완료", "해결됨" 등 작업이 이미 끝났음을 명시하는 경우.

## 3. Gemini API 프롬프트 템플릿

위의 논리를 바탕으로 Gemini API에 전송할 프롬프트 시스템 지시어(System Instruction) 및 사용자 메시지 형식입니다.

### 3-1. 시스템 프롬프트 (System Prompt)

```text
당신은 사용자의 할 일(Task)을 분석하여 적절한 속성으로 분류해주는 숙련된 AI 칸반보드 매니저입니다.
사용자가 할 일의 제목과 설명을 제공하면, 다음 기준에 따라 분석하여 오직 JSON 형식으로만 응답해야 합니다.

[분류 기준]
1. category (문자열): 업무의 성격을 파악하여 다음 중 하나로 분류하세요.
   [개발, 디자인, 마케팅, 기획, 운영, 기타] 중 가장 적합한 하나를 선택합니다.
2. priority (문자열): 업무의 중요도와 긴급도를 파악하여 다음 중 하나로 분류하세요.
   - high: 긴급, 오늘까지, 치명적 버그, 장애 해결 등
   - medium: 일반적인 업무 (기본값)
   - low: 나중에, 여유있을 때, 단순 아이디어 등
3. status (문자열): 업무의 진행 상태를 파악하여 다음 중 하나로 분류하세요.
   - todo: 해야 할 일, 계획 중인 일 (기본값)
   - in_progress: 현재 진행 중, 작성 중, 검토 중인 일
   - done: 이미 완료된 일, 끝난 일

[응답 형식]
설명이나 추가 텍스트 없이 오직 유효한 JSON 객체 하나만 반환하세요.
{
  "category": "분류된 카테고리명",
  "priority": "high | medium | low",
  "status": "todo | in_progress | done"
}
```

### 3-2. 사용자 요청 프롬프트 예시 (User Prompt)

```text
[할 일 정보]
제목: 인스타그램 광고 캠페인 기획하기
설명: 여름 시즌 대비 3가지 버전 카피 작성 필요
```

### 3-3. 기대되는 JSON 응답 예시 (Expected Output)

```json
{
  "category": "마케팅",
  "priority": "medium",
  "status": "todo"
}
```

## 4. 예외 및 실패 처리 로직

Gemini API가 예상과 다른 형식으로 응답할 경우를 대비한 클라이언트(혹은 백엔드) 측 보완 로직입니다.

- **JSON 파싱 실패**: 응답 텍스트에서 앞뒤 마크다운(` ```json ` 등)을 제거하는 전처리 후 파싱 시도. 그래도 실패 시 해당 카드의 AI 분석을 실패 처리(재시도 유도).
- **잘못된 값 반환**: `priority`나 `status` 필드에 지정된 Enum 값(high/medium/low, todo/in_progress/done)이 아닌 다른 문자열이 오면, 각각의 기본값인 `medium`과 `todo`로 강제 변환(Fallback).
- **부분 실패 허용**: 여러 카드를 일괄 분석(AI 정리)할 때, 특정 카드 하나가 분석에 실패하더라도 다른 카드의 분류 및 이동 결과에는 영향을 주지 않도록 각각 독립적으로 처리. (PRD FR-05 요구사항)
